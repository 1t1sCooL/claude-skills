# Outreach Tracker

Реестр холодных рассылок и фоллоу-апов: кому писал, когда, статус, дата следующего касания.
Низ воронки после `frontend-audit` → `proposal`. Цель — чтобы ни один лид и ни один фоллоу-ап
не потерялись. Тон шаблонов — конструктивный, через ценность, без «купи».

Реестр живёт в `./outreach/outreach.md` (человекочитаемо) + `./outreach/outreach.csv`
(данные: `lead, url, channel, status, last_touch_date, next_followup_date, notes`).
Статусы: `new → contacted → replied → in_talks → won / lost / cold`.

## Что на входе (`$ARGUMENTS`)

Подкоманда:
- **`add <url|имя>`** — завести лид (спросить недостающее: канал, заметка), статус `new`.
- **`touch <lead>`** — зафиксировать касание: проставить `last_touch_date` = сегодня, сдвинуть `next_followup_date` (по умолчанию +3 дня для 1-го фоллоу-апа, +7 для 2-го), обновить статус.
- **`due`** — показать, кого пора пинговать (next_followup ≤ сегодня) + просроченные.
- **Пусто** — сводка: сколько лидов по статусам, ближайшие касания, кто завис.

> Сегодняшнюю дату бери из окружения (системная дата сессии).

## Шаги

1. **Прочитать реестр** (`./outreach/outreach.csv`); если нет — создать с заголовком.
2. **Выполнить подкоманду**, обновить обе версии (`.csv` — источник правды, `.md` — рендер для чтения).
3. **При `add`/`touch`** — предложить подходящий шаблон касания (см. ниже) под текущий этап.
4. **При `due`/пусто** — отсортировать по срочности, явно выделить просроченные.

## Шаблоны касаний

- **Первичное** — зацепка из мини-аудита: «заметил <конкретная проблема> на <сайт>, это влияет на <следствие>; если интересно — покажу, что и как чинится». Коротко, конкретно, без воды.
- **Фоллоу-ап 1 (через 3–4 дня)** — добавить ценность (ещё одно наблюдение / пример), мягкое «актуально ли».
- **Фоллоу-ап 2 (через неделю)** — финальное: «закрываю тему с моей стороны, если что — вот как связаться»; без давления, оставить дверь открытой.

## Выход

- Обновлённые `./outreach/outreach.md` + `.csv`.
- Что делать дальше: кого пингануть сегодня и каким шаблоном.
